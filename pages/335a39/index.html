# 跨域

  • 为什么会产生跨域
    • 基于浏览器同源策略即协议、主机、端口号不相同就会产生跨域
    • 浏览器防止不同站点之间随意通信，造成安全问题，比如XSS或者CSRF
  • 解决方案
    • cors
    • jsonp
    • websocket
    • proxy
      • nginx
      • config.js文件配置proxy
    • postMessage

# http1.1

1. 新增了缓存策略

    由首部Transfer-Encoding控制传输编码，可设置chunks表示分块传输
5. 新增host请求头
    表示接收请求的服务器的主机名和端口

# http2.0

1. 支持多路复用
    http请求和响应不必按顺序执行，避免队头阻塞
2. 二进制分帧
    数据在传输过程中采用二进制帧的格式封装，在帧的头部会有一个流标识，可作为合成标志
3. 头部压缩
    客户端和服务端维护一张头部字段的表，避免重复数据进行发送，每次发送请求会在表中查找差异数据
4. 支持服务端推送资源
    对于请求页面一些可能需要用到的资源，如样式、js等资源会一并推送给客户端

# websocket

  • 定义：属于应用层上的全双工通信网络技术，可建立在单个tcp连接上的网络传输协议
  • 对比http：
    • 请求url以ws或者wss开头
    • 在htp1.1之前不能进行长连接，而ws只需要经过一次握手，便能够建立持久性的连接，且能够进行双向通信
    • 允许服务器主动向客户端推送消息
  • 应用场景
    • 聊天室
    • 实时数据
      • 在线协同编辑
      • 多人游戏
      • 实时地图

# http缓存

  • 用户请求资源过程

    • 判断浏览器是否有强缓存
    • 如果有，就判断强缓存是否过期
      • 如果没有过期，就从资源缓存中找
      • 如果过期，就判断是否存在响应头ETag
        • 如果有，向服务器请求，并携带请求头If-none-match，根据服务端返回200还是304，返回资源
        • 如果没有，就判断是否存在请求头Last-modified
          • 如果有，向服务器请求，携带请求头If-modified-since，根据服务返回200还是304，返回资源
          • 如果没有，直接向服务器发送请求，响应请求后，再进行缓存协商，返回资源
    • 如果没有，直接向服务器请求资源，响应内容后再缓存协商，返回资源

  • 强缓存

    • Expires
    • Cache-Control

  • 协商缓存

    • Last-Modified、 If-Modified-Since
    • ETag 、If-None-Match

  • 区别

    • 强缓存命中，不会请求服务器，直接请求缓存
    • 协商缓存命中，会请求服务器，不会返回内容，然后读取缓存
    • 利用缓存，可以有效减少网络流量，减轻服务器压力

# http状态码

  • 1xx 信息类状态码
  • 2xx 成功类状态码
  • 3xx 重定向状态码
  • 4xx 客户端错误装代码
  • 5xx 服务端错误状态码

# http请求方法

  • 简单请求
    • 请求方式为HEAD、POST 或者 GET
  • 复杂请求
    • 请求方法是PUT或DELETE，或者Content-Type字段的类型是application/json

# get请求和post请求方法

  • get请求是幂等的，post是非幂等的
    • 幂等：一次请求和多次请求的结果包括http状态是不会改变的
    • get请求一般不会对服务器造成影响，而post请求一般用来创建资源
  • get请求会被浏览器缓存，post请求一般不会
  • get请求数据格式有限，post请求支持form表单和文件格式等多种复杂数据

# 三次握手

  • 第一次握手发送SYS包，与服务端建立连接
  • 第二次握手发送SYS+ACK包，返回客户端应答连接
  • 第三次握手发送ACK包给服务端，表示已经建立可靠连接
  • 二次握手
    • 如果遇到网络阻塞问题，在经过第一次发送SYS包建立连接时，会发送不出去
    • 重发SYS包后，等到第二次握手时，假如网络变好了，第一次发送的SYS包继续发送，这时服务器会认为这是两次连接
    • 其实客户端只发了一次连接，造成状态不一致，不可靠的情况，也浪费了服务器资源
  • 三次握手失败后会如何
    • 服务端由于没有收到ACK包，会重新发送SYS+ACK包，次数默认5次，后续如果还是没有，就关闭连接
    • 但是客户端认为已经建立连接，后续发送请求，服务端会以RST包作为响应，重置连接

# 四次挥手

  • 第一次挥手，客户端向服务器发送FIN包，表示关闭连接，此时状态为终止等待状态1FIN_WAIT1

  • 第二次挥手，服务端接收到FIN包，会给客户端发送ACK包，表示自己收到报文，服务端进入关闭等待状态CLOSE_WAIT，客户端收到后，进入终止等待状态2FIN_WAIT2

  • 第三次挥手，此时服务端也想断开连接，需要给客户端发送FIN包，服务端进入最后确认状态LAST_ACK，等待客户端确认

  • 第四次挥手，客户端收到FIN包，也会发送ACK包作为应答，此时客户端处于时间等待状态TIME_WAIT，需要等到2MSL，才释放TCP连接，需要经过2MSL后，等到服务端收到ACK包，才属于关闭状态CLOSED

  • 为什么等到2MSL

    • 2MSL指的是最大段生存时间，是说网络中存在的报文超过一定时间，就会超时被抛弃
    • 为什么？防止网络波动导致丢包问题，需要等到2MSL后，服务端接收到客户端发送的ACK确认报文后才处于CLOSED状态

  • 为什么需要四次挥手

    • 在关闭连接时，服务端收到客户端发送的FIN包后，可能不会立即关闭连接，所以只能先发送一个ACK包，告诉客户端自己收到FIN包了，等我所有包发送完毕后才能发送FIN包，不能一次性发送，所以需要四次挥手

  • 怎么判断重传

    • TCP为了保证数据在网络波动情况下，能够提高可靠性传输服务，提出重传机制
    • 开启一个定时器，在这段时间内，没有收到ACK确认报文，就进行重传，如果还是没有，就会放弃，并发送一个复位信号报文

# TCP和UDP的区别

  • TCP面向连接，UDP是非连接的（发送数据不需要建立连接）
  • TCP是可靠的，UDP是不可靠的
    • UDP超时不会重传报文
    • TCP适合大数据的传输
  • TCP的传输效率比较慢，UDP的传输效率比较快
  • TCP面向字节流，UDP面向报文
    • UDP容易造成丢包问题
  • TCP只能支持点对点全双工通信，UDP支持一对一、一对多、多对一、多对多等多种方式通信
